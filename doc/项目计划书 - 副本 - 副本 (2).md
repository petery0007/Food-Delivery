## 一、项目概述

### 1. 项目背景
随着人们对健康饮食的重视，绿色有机食品需求日益增长。然而传统线下购买存在信息不透明、渠道有限、配送效率低等问题。开发一个基于WEB的绿色有机食品配送系统，可以利用网络实现商品展示、在线下单和基础管理，解决信息零散、购买不便的问题，提升用户体验和管理效率。

### 2. 项目目标
- 实现用户注册、登录、浏览商品、加入购物车、下单（模拟支付）、查看订单等基本购物功能。
- 实现管理员对商品信息（增删改查、上下架）和订单（发货、状态更新）的管理。
- 实现配送人员对订单的接取与派送。
- 保证基本的数据安全性（密码加密、权限控制）和响应性能。
- 在四周内交付一个可演示、可运行的原型系统。

### 3. 开发环境与条件
- **开发工具**：IntelliJ IDEA 
- **后端技术**：Spring Boot + MyBatis-Plus  
- **前端技术**：Vue2 + Element UI  
- **数据库**：MySQL 8.0  
- **服务器**：内置Tomcat  
- **运行环境**：Windows，Chrome浏览器

## 二、需求分析

### 1. 目标用户与使用场景
| 用户角色 | 使用场景 |
|---------|----------|
| 普通用户 | 注册登录后，浏览商品列表，按分类或关键词搜索，查看商品详情，加入购物车，下单（模拟支付），查看自己的订单状态。 |
| 管理员 | 登录后台，管理商品（新增、编辑、删除、上下架），管理用户订单（查看所有订单、修改订单状态为“已发货”/“已完成”），管理普通用户账号。 |
|配送人员|查看配送订单，明确配送商品信息，在送达后更新配送订单状态。|

### 2. 核心功能清单
- **用户端**  
  - 注册、登录、退出  
  - 商品列表展示（分页、按分类筛选、关键词搜索）  
  - 商品详情页（名称、图片、价格、库存、简介）  
  - 购物车（添加商品、修改数量、删除商品、计算总价）  
  - 下单（选择收货地址、生成订单、模拟支付成功）  
  - 我的订单（查看所有订单，按状态筛选）  
  - 个人资料修改、收货地址管理  

- **管理员端**  
  - 商品管理（查看商品列表、新增商品、编辑商品、删除商品、上下架）  
  - 订单管理（查看所有用户订单、修改订单状态为“已发货”/“已完成”）  
  - 用户管理（查看用户列表、禁用/启用用户、重置密码）  
  - 公告管理（发布、编辑系统公告）
- **配送员端**
- 订单信息查询（配送商品地址，接收用户等等）
- 配送状态更新（已配送，未配送）

### 3. 非功能需求
- **性能要求**：商品列表加载时间 ≤ 2秒，订单提交响应 ≤ 3秒。  
- **安全要求**：密码使用BCrypt加密，接口需登录验证，普通用户无法越权操作他人订单。  
- **兼容性要求**：适配Chrome、Edge最新版本，PC端显示正常。

## 三、技术路线

### 1. 总体架构
前后端分离：
- 前端：Vue2 + Element UI + Axios  
- 后端：Spring Boot + MyBatis-Plus + JWT（身份认证）  
- 数据库：MySQL 8.0  

### 2. 主要技术
| 层次       | 技术选型                          |
|------------|-----------------------------------|
| 前端框架   | Vue2, Vue Router, Vuex, Axios     |
| UI组件库   | Element UI                        |
| 后端框架   | Spring Boot 2.7, MyBatis-Plus     |
| 安全       | JWT, BCrypt                       |
| 数据库     | MySQL 8.0                         |
| 工具       | Maven, Git, Postman               |

### 3. 数据设计初步考虑
- **主要实体**：用户(user)、商品(product)、订单(order)、订单项(order_item)、收货地址(address)、公告(notice)。  
- **关键数据表**：  
  - `user`：id, username, password(加密), phone, email, role, status  
  - `product`：id, name, category, price, stock, image_url, description, status  
  - `order`：id, order_no, user_id, total_amount, status, address, create_time  
  - `order_item`：id, order_id, product_id, product_name, price, quantity  
  - `address`：id, user_id, receiver_name, phone, detail, is_default  
  - `notice`：id, title, content, create_time  
- **数据关系**：用户与订单一对多；订单与商品多对多（通过order_item）。

### 3. 风险与应对
| 风险 | 应对策略 |
|------|----------|
| 时间紧张，功能完成度不足 | 优先保证核心购买流程（商品→购物车→下单→订单查看）；管理员基础管理够用即可。 |
| 前后端联调问题 | 提前约定API接口格式，使用Swagger生成文档，后端先提供mock数据。 |
| 购物车状态丢失 | 要求用户登录后才可使用购物车，购物车数据保存在后端数据库。 |
| 支付实现复杂 | 模拟支付：点击“提交订单”后直接设置订单状态为“已支付”。 |
